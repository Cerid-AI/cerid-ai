## Overview

Phase 7 builds on the complete React GUI and hardened backend from Phase 6 to add intelligent automation layers. The focus shifts from infrastructure and UI to making Cerid AI proactively useful — detecting errors in LLM output, optimizing model selection for cost, and automatically growing the knowledge base from conversations and local files.

Three sub-phases, each independently deployable:

| Sub-Phase | Focus | Key Deliverables |
|-----------|-------|-----------------|
| **7A** | Audit Intelligence | Hallucination detection, conversation analytics, enhanced feedback loop |
| **7B** | Smart Orchestration | Model router, auto-switch, cost dashboard with budget alerts |
| **7C** | Proactive Knowledge | Drive scanning, memory extraction, smart KB suggestions |

## 7A: Audit Intelligence

### Hallucination Detection Agent

**Goal:** Differential analysis comparing LLM assertions against KB facts in real-time.

**Architecture:**
- New agent: `agents/hallucination.py`
- Post-response hook: after each LLM response, extract factual claims
- Cross-reference claims against KB artifacts (vector similarity + keyword match)
- Score each claim: `verified` (KB match found), `unverified` (no data), `contradicted` (KB disagrees)
- Results surfaced in the audit tab as a confidence-scored fact-check panel

**Implementation details:**
- Claim extraction via lightweight LLM call (Llama 3.1 free tier) — extract JSON array of factual statements
- KB verification uses existing `/agent/query` endpoint with high `top_k` for recall
- Similarity threshold: 0.75 for `verified`, <0.4 for `unverified`
- Contradictions detected via entailment check (LLM judges if KB content supports or contradicts the claim)
- Results cached in Redis with conversation ID linkage

**React GUI integration:**
- New component: `components/audit/hallucination-panel.tsx`
- Shows per-message fact-check results inline or in audit pane
- Color-coded: green (verified), yellow (unverified), red (contradicted)
- Click-through to source KB artifacts

### Conversation Analytics

**Goal:** Track model usage, cost per conversation, and token efficiency across models.

**Architecture:**
- Extend `agents/audit.py` with conversation-level metrics
- Track per-conversation: model used, input/output tokens, cost, response latency
- Aggregate by model, time period, domain affinity
- Store in Redis with structured keys: `conv:{id}:metrics`

**React GUI integration:**
- Extend chat-dashboard.tsx with session-level cost tracking
- New audit sub-tab: "Conversations" — sortable table of past conversations with cost/token columns
- Model comparison charts: cost-per-token, avg response quality, latency distributions

### Enhanced Feedback Loop

**Goal:** Auto-extract facts and code from chat responses and store as KB artifacts.

**Architecture:**
- Post-response extraction pipeline (opt-in via `ENABLE_FEEDBACK_LOOP`)
- Extract structured content: code blocks, factual statements, decisions, preferences
- Create new KB artifacts in a `conversations` domain
- Bidirectional graph links: conversation artifact ↔ referenced KB artifacts
- Dedup against existing KB content (content_hash check)

**Implementation details:**
- New domain: `conversations` (add to `config.DOMAINS`)
- Extraction uses Llama 3.1 to identify extractable content types
- Code blocks extracted verbatim with language tag as metadata
- Facts extracted as concise statements with source conversation ID
- Graph relationships: `EXTRACTED_FROM` (conversation → artifact), `REFERENCES` (artifact → KB source)
- Respect token budget: skip extraction for short responses (<100 tokens)

## 7B: Smart Orchestration

### Model Router

**Goal:** Client-side intelligence that calculates optimal model for each query based on cost vs complexity.

**Architecture:**
- New module: `src/web/src/lib/model-router.ts`
- Input signals: query length, detected domain, conversation depth, KB injection count, user preference
- Output: recommended model with cost estimate and reasoning
- Factors considered:
  - **Context window utilization** — longer conversations benefit from larger context models
  - **Input/output pricing** — OpenRouter per-model pricing table (cached)
  - **Domain affinity** — coding queries → Claude/GPT, simple queries → Llama/Gemini Flash
  - **Conversation length** — short exchanges can use cheaper models
  - **Context switch cost** — summarizing conversation for new model context has a token cost

**Implementation details:**
- Model pricing table fetched from OpenRouter API, cached in localStorage (24h TTL)
- Complexity scoring: simple heuristics first (query length, keyword detection), LLM classification later
- Domain detection reuses Bifrost's intent classifier
- Configurable via SettingsContext: `autoModelSwitch` (boolean), `costSensitivity` (low/medium/high)

### Auto-Switch Recommendations

**Goal:** When a cheaper model can handle the query equally well, suggest or auto-switch.

**Architecture:**
- Pre-send hook in `use-chat.ts`: before sending message, evaluate model router recommendation
- If current model is significantly more expensive than recommended, show suggestion banner
- Auto-switch mode: silently switch for clear cost wins (>3x price difference, same capability tier)
- Manual mode: show toast notification with cost comparison, user clicks to accept

**React GUI integration:**
- Model recommendation banner above chat input
- "Switch to [model] — save ~$X.XX" with one-click accept
- Settings toggle: auto-switch on/off, cost sensitivity slider

### Cost Dashboard

**Goal:** Real-time session cost tracking with budget alerts and model comparison.

**Architecture:**
- Extend chat-dashboard.tsx with running cost accumulator
- Budget system: user sets daily/weekly/monthly budget in settings
- Alert thresholds: 75% (yellow warning), 90% (orange), 100% (red, optional auto-pause)
- Model comparison panel: side-by-side cost/quality for recent conversations

**Implementation details:**
- Cost calculated from token counts + model pricing table
- Stored in localStorage per session, aggregated in Redis for historical tracking
- Budget alerts via toast notifications
- New settings page section: "Budget & Cost" with threshold configuration

## 7C: Proactive Knowledge

### Drive Scanning

**Goal:** Auto-ingest from configurable directories beyond `~/cerid-archive`.

**Architecture:**
- Extend `watch_ingest.py` with configurable watch paths
- New config: `WATCH_PATHS` — JSON array of `{path, domain, mode, recursive}` objects
- Support Google Drive sync folders, Downloads, Desktop, custom paths
- Incremental scanning: track last-modified timestamps, only process changed files
- Ignore patterns: `.git/`, `node_modules/`, `__pycache__/`, configurable via `WATCH_IGNORE`

**Implementation details:**
- Config stored in `.env` as `WATCH_PATHS='[{"path":"/Users/x/Downloads","domain":"general","mode":"smart"}]'`
- Watchdog library already used — extend with multi-path support
- Rate limiting: max 10 files/minute to avoid overwhelming the ingestion pipeline
- File size limit: skip files >50MB
- Duplicate detection: content_hash check before full ingestion

### Memory Extraction

**Goal:** Parse chat conversations for facts, decisions, and preferences. Auto-create KB entries.

**Architecture:**
- Post-conversation hook (triggered on conversation end or after N messages)
- LLM extraction prompt identifies: facts learned, decisions made, preferences expressed, action items
- Each extraction becomes a KB artifact in the `conversations` domain
- Memory dedup: check against existing conversation artifacts before creating
- Graph links to any KB artifacts referenced during the conversation

**Implementation details:**
- Extraction runs asynchronously (background task) to avoid blocking chat
- Uses Llama 3.1 for extraction (cost-free)
- Memory types tagged in metadata: `memory_type` = `fact|decision|preference|action_item`
- Retention policy: memories older than 6 months auto-archived (not deleted)
- User can review/edit/delete extracted memories in KB pane

### Smart Suggestions

**Goal:** Proactively surface relevant KB content based on conversation patterns.

**Architecture:**
- Real-time conversation analysis: as user types, identify topics and intent
- Background KB query with low latency requirements
- Suggestion panel in KB context pane: "You discussed this before — here's what you said"
- Temporal awareness: recent conversations weighted higher
- Cross-domain discovery: surface related content from unexpected domains

**Implementation details:**
- Debounced analysis: trigger after 500ms of typing pause (not on every keystroke)
- Use existing `/agent/query` with conversation context as query
- Filter to exclude already-injected artifacts
- Rank by recency + relevance blend (configurable weights)
- Max 3 suggestions shown at a time, dismissible
- "Pin" feature: user can pin a suggestion to keep it visible

## Implementation Order

Recommended sequence: **7A → 7B → 7C**

- 7A (Audit Intelligence) builds on existing audit infrastructure and provides immediate value
- 7B (Smart Orchestration) requires conversation analytics from 7A for informed model routing
- 7C (Proactive Knowledge) is the most ambitious and benefits from the cost optimization in 7B
